# Milestone 05 — UI Release

## Overview

Milestone 05 introduces a browser-based **Control Panel** for OptiMoldIQ — allowing non-technical users to interact with the system without writing code. The panel is built with React + Vite, served directly by the existing FastAPI backend, and accessible via a shareable URL through Google Colab + ngrok.

The core system (workflow engine, modules, dependency policies) is unchanged. This milestone is purely additive — a UI layer on top of the M04 framework.

---

## What Changed

### New: Control Panel (`control_panel/`)

The control panel is a React single-page application compiled to static files and served by FastAPI from `control_panel_dist/`.

```
control_panel/
├── index.html                    ← Vite HTML entry point
├── package.json                  ← Node dependencies & build scripts
├── vite.config.js                ← Vite bundler config
├── src/
│   └── main.jsx                  ← React app entry point (NEW)
├── App.jsx                       ← Root component (unchanged)
├── data.js                       ← Static reference data (unchanged)
├── analytics_module/             ← Analytics views
├── initial_planning_module/      ← Initial planning views
└── progress_tracking_module/     ← Progress tracking views
```

**What's new in this milestone:**

`index.html`, `package.json`, `vite.config.js`, and `src/main.jsx` are the four files added to make the existing JSX components buildable as a proper Vite project. The component code (`App.jsx`, `data.js`, and the three module folders) carried over from earlier development without modification.

**Build output:**

Running `npm run build` inside `control_panel/` compiles the React app to `control_panel_dist/` — a folder of static HTML, JS, and CSS files. This folder is gitignored and must be built before starting the server.

---

### New: FastAPI Static File Serving (`api/server.py`)

FastAPI mounts `control_panel_dist/` as a static directory and serves it at the root path. No separate frontend server is needed — the same `uvicorn` process serves both the API and the UI.

```
GET /          → control_panel_dist/index.html  (Control Panel)
GET /api/...   → FastAPI routes (workflows, execution, viz)
```

---

### New: Colab Launcher Notebook (`control_panel_notebook.ipynb`)

A self-contained notebook that handles the full setup sequence for non-technical users:

1. Clone (or pull) the repository
2. Install Python dependencies
3. Build the control panel (`npm install` + `npm run build`)
4. Launch the FastAPI server and expose it via ngrok

ngrok authentication is read from Colab Secrets — users add their token once via the Colab UI (🔑 sidebar) and never need to touch the code.

---

## Control Panel Modules

The UI is organized around three functional modules that map directly to the workflow domains in the backend:

**Initial Planning Module** — configure and trigger initial production planning workflows. Exposes workflow parameters and displays structured output on completion.

**Progress Tracking Module** — monitor active and completed workflow executions. Shows execution status, module-level results, and dependency resolution outcomes.

**Analytics Module** — explore production analytics and dashboards generated by the analytics workflows. Visualizations are rendered from data returned by the `/api/viz` routes.

## How to Run

### Via Google Colab (recommended for non-technical users)

1. Open `control_panel_notebook.ipynb` in Google Colab
2. Add your ngrok authtoken to Colab Secrets (key name: `NGROK_TOKEN`)
3. Runtime → Run All
4. Click the URL printed in the final cell

### Locally

```bash
# Build the UI first (one-time, or after UI changes)
cd control_panel
npm install
npm run build
cd ..

# Start the server
python main.py

# Open in browser
open http://localhost:8000
```

## Constraints & Known Limitations

**Session-bound URL** — The ngrok public URL changes on every Colab session restart. A new URL must be shared with the team after each startup.

**Single-user Colab session** — The Colab runtime runs on one machine. Multiple users can access the app via the shared URL simultaneously, but the session is tied to whoever started the notebook.

**Build required before serving** — `control_panel_dist/` is gitignored. Anyone running locally must run `npm run build` before starting the server. The Colab notebook handles this automatically.

**No persistent storage** — Data files are read from the repository. New monthly Excel files should be committed to the repo; the notebook pulls the latest on startup via `git pull`.
